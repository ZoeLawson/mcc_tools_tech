What is an API

Application Programming Interface

A way for blobs of code to talk to each other.

A blob of code can be an app, a web site, a service, whatever.

Currently, very 'in' in the software world, thanks to microservices.

In the olden days...

Once upon a time, people created huge, monolithic software applications.

A large team of developers work on different components of the software, making sure everything talks to each other.

Basically, a giant black boxes. Input data, get 'sausage' out the other side, with zero way to get inside the "black box".

Building/compiling the software takes hours.

Reworking the software very difficult.

Enter micro-services

Make a small black box and hook together the boxes.

Small teams work on the small boxes.

Take in data (via an API), manipulate it, spit out data (via an API)

Smaller boxes easier to code, test, and update.

Anyone can use any box at any time as long as you supply data in the right format.

Internal vs External

Most companies need both internal and external APIs

  • Internal — not meant to be shared with anyone outside of the development team.
  • External (or Open) — available to anyone who wants to interact with the software.

Tech writers should be involved with documenting external APIs.

Being involved with internal APIs depends on the company.

Types of APIs

There are two major types of APIs:

  • Non-web APIs
  • Web APIs

There are many, many non-web based APIs hanging around. These can be based on any programming language, such as C++ or Java.

Currently, a lot of APIs are a web-based API (because everyone loves hosted services).

Web APIs

There are many, but there are two major flavors today

  • SOAP (Simple Object Access Protocol) — XML based. Very structured, very heavy. Uses a WSDL (Web Services Definition Language)
  • REST (Representational State Transfer) — A set of architectural principles.

There's also the older XML- and JSON-RPC (Remote Procedure Calls) protocols.

REST APIs

For the non-technical person, i.e. what I've figured out using them.

Using URLs, you can send or request data. Overly simplified, you can

  • GET — An API call that ususally retrieves information. You may have to provide some data (e.g. an object ID) to get specifics.
  • POST — Create (or update) a thing
  • PUSH — Update (or create) a thing
  • DELETE — Remove a thing

How to document APIs

All depends on your relationship with development/engineers

  • If developers don't want to work with you at all, use your regular authoring tools
  • If you get a little cooperation, have them generate output that you massage.
  • If you get a lot of cooperation:
    • You can edit comments in the source for generating output.
    • Developers can write some of the documentation.

Minimum requirements

You have to include a referenced of all the calls/endpoints

  • Name of the API
  • What data you need to send.
  • What data (response) you expect to get in return.
  • Common errors.
  • If at all possible, an example

This content can be generated out of source.

Other API content

There is other API content you should include:

  • Recommended architecture
  • How to authenticate and/or encrypt data, and other security concerns
  • How to get started
  • Common use cases

And if you're really lucky, an environment to test with.

Tips for documenting APIs

  • Just because it's written for developers, don't assume developers are your only audience.
  • Don't assume your developers are experienced.
  • As with all content, provide use case scenarios and examples, as well as reference.

Swagger and OpenAPI

A framework for providing documentation for REST APIs that can also be interactive.

Can be generated from source code or with a JSON or YAML file.

If generating from source code, depends on what the source is authored in. See this article for a list of options per language.
